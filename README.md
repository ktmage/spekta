# Spekta

**機械的に検証可能なコードから、人間が読めるドキュメントを生成する — AI 時代のドキュメンテーションフレームワーク**

## 概要

Spekta は、機械的に正しさを検証可能なコード（テストコード、スキーマ定義、ルーティング定義など）を情報源として、人間が読めるドキュメントを自動生成するツールチェインです。

AI がコードを大量に生成する時代において、人間はアプリケーションをより高い抽象度で可視化・把握する必要があります。テストコードは AI にとって既に最良のドキュメントです — 検証可能で、曖昧さがなく、開発を駆動する唯一の信頼源です。Spekta はこれを**人間向けのドキュメントに変換する仕組み**です。

## モチベーション

- **AI 時代の可視化**: AI が生成したコードの品質を、人間が全行レビューするのではなく、振る舞いの検証に集中する。人間はアプリケーションの全体像をより高い抽象度で把握する必要がある
- **機械的に検証可能 = ドキュメントの源泉**: テストを走らせれば振る舞いの正しさがわかる。スキーマを見ればデータ構造の正しさがわかる。このように機械的に正しさを確認できるコードこそが、ドキュメントの信頼できる源泉になる
- **テストが true = 仕様である**: テストに書かれていない振る舞いは仕様書にも載らない。仕様が足りなければテストを書く。この一貫したサイクルでドキュメントの鮮度を保つ

## 基本思想

### `--format documentation` の延長線上

RSpec の `--format documentation` は、テストの階層構造を人間が読めるドキュメントとして出力する。Spekta が生成するコンテンツはこれと同質だが、以下の体験上の差を埋める:

- **テストを実行しない**: 静的解析によりコードがあれば生成できる。実行環境や DB が不要
- **永続的な Web UI**: ナビゲーション・検索が可能な静的サイトとして出力する
- **コメント・メタデータの反映**: `it` の文字列だけでなく、テストコード内の補足情報も仕様書に反映する
- **ファイル横断の構造**: テストスイート全体を機能マップとして俯瞰できる

### 機能の重なりは問題ない

各 `describe` はその機能が何を担保しているかを表明しているだけであり、複数の機能間で振る舞いが重なっていても問題ない。それぞれの機能の視点から書かれた仕様として自然に共存する。ある機能の改修で別の機能のテストが落ちるなら、それはデグレ検知として正しい。

### 「なぜ」の補足（検討中）

テストコード自体は「何が・どの条件で・どう動くか」を表現する。「なぜそうなのか」を補う手段として以下を検討:

- **コメント / メタデータ**: テストコード内にメタデータ的なコメントを記述する（RBS inline 的な記法など）
- **ヘルパーメソッドによる拡張**: テストフレームワークの拡張で「なぜ」を書く場所を提供する（理想的だが保守コストとのトレードオフ）

## ドキュメントの種類

開発に必要なドキュメントを以下のように分類する。Spekta はこのうち、機械的に検証可能なコードから生成できるものを対象とする。

| 種類 | 内容 | コードから生成 |
|---|---|---|
| 振る舞い仕様 | このシステムは何をするのか | テストコードから生成可能 |
| データ構造 | どんなデータを持っているのか | スキーマ定義から生成可能 |
| インターフェース定義 | 外部・内部とどうつながるのか | ルーティング定義等から生成可能 |
| アーキテクチャ | なぜこの構造にしたのか | アーキテクチャテストから部分的に可能（言語特性に依存） |
| 運用 | どう動かし、監視し、直すのか | コードから生成困難（対象外） |
| オンボーディング | 新しい人がどう始めるのか | コードから生成困難（対象外） |

## アーキテクチャ

```
[アナライザー群]                [IR群]                [出力]

System spec アナライザー  ─┐
Feature spec アナライザー ─┼→ 振る舞いIR ─────┐
Request spec アナライザー ─→ API IR ────────┼→ Web UI
Model spec アナライザー  ──→ データモデルIR ─┘
```

3 つのレイヤーで構成され、それぞれが抽象化されている。

### 1. アナライザー（コード → IR）

コードを静的に解析し、種類ごとの IR に変換する。

- テストを実行せずにコードを解析する
- ドキュメントの種類ごとにアナライザーが独立しており、環境に応じて付け外しできる
- RSpec においては spec の種類（System / Feature / Request / Model など）がドキュメントの種類と自然に対応する
- 同じ IR に対して複数のアナライザーが存在しうる（例: System spec と Feature spec は DSL が異なるが、どちらも振る舞い IR を出力する）

### 2. IR（中間表現）

ドキュメントの種類ごとに独立した中間表現。データの形が異なるため、単一の IR にはしない。

- **振る舞い IR**: ツリー構造（describe > context > it）
- **データモデル IR**: エンティティと関連（テーブル、カラム、リレーション）
- **API IR**: エンドポイント、メソッド、パラメータ

IR 間は ID や名前の一致で緩くリンクでき、Web UI 上で相互参照が可能。

### 3. ドキュメント生成レイヤー（IR → Web UI）

複数の IR を統合し、人間が読みやすい Web ドキュメントを生成する。

- 各 IR の内容を一つの静的サイトに統合して表示
- ナビゲーション・検索が可能
- 開発中はホットリロード対応、ビルドすれば静的サイトとしてデプロイ可能

## 第一段階の方針

- **System/Feature spec アナライザー + 振る舞い IR + Web UI** の一本道を通す
- System spec と Feature spec はアナライザーを分けて、出力先は同じ振る舞い IR とする
- パイプラインの構造さえ確立すれば、後から別のアナライザーと別の IR を足して Web UI に繋ぎ込むだけで拡張できる

## 名前の由来

**Spekta** — 「スペクタ」はラテン語の *spectare*（観察する、見る）に由来し、英語の *spec*（仕様）と *spectra*（スペクトル、光の分解）を連想させます。テストコードという光をプリズムのように分解し、仕様書として可視化する、というコンセプトを表現しています。
