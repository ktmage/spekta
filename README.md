# Spekta

**テストコードから仕様書を生成する — AI 時代のドキュメンテーションフレームワーク**

## 概要

Spekta は、テストコード（RSpec）を唯一の信頼できる情報源（Single Source of Truth）として、人間が読める仕様書を自動生成するツールチェインです。

AI がコードを大量に生成する時代において、人間はアプリケーションをより高い抽象度で可視化・把握する必要があります。テストコードは AI にとって既に最良のドキュメントです — 検証可能で、曖昧さがなく、開発を駆動する唯一の信頼源です。Spekta はこれを**人間向けのドキュメントに変換する仕組み**です。

## モチベーション

- **AI 時代の可視化**: AI が生成したコードの品質を、人間が全行レビューするのではなく、振る舞いの検証に集中する。人間はアプリケーションの全体像をより高い抽象度で把握する必要がある
- **テストコード = ドキュメントの源泉**: テストは振る舞いの正しさを機械的に証明できる唯一の成果物。AI にとってのドキュメントであるテストコードを、人間にとってのドキュメントにも変換する
- **テストが true = 仕様である**: テストに書かれていない振る舞いは仕様書にも載らない。仕様が足りなければテストを書く。この一貫したサイクルでドキュメントの鮮度を保つ

## 基本思想

### `--format documentation` の延長線上

RSpec の `--format documentation` は、テストの階層構造を人間が読めるドキュメントとして出力する。Spekta が生成するコンテンツはこれと同質だが、以下の体験上の差を埋める:

- **テストを実行しない**: 静的解析によりコードがあれば生成できる。実行環境や DB が不要
- **永続的な Web UI**: ナビゲーション・検索が可能な静的サイトとして出力する
- **コメント・メタデータの反映**: `it` の文字列だけでなく、テストコード内の補足情報も仕様書に反映する
- **ファイル横断の構造**: テストスイート全体を機能マップとして俯瞰できる

### 機能の重なりは問題ない

各 `describe` はその機能が何を担保しているかを表明しているだけであり、複数の機能間で振る舞いが重なっていても問題ない。それぞれの機能の視点から書かれた仕様として自然に共存する。ある機能の改修で別の機能のテストが落ちるなら、それはデグレ検知として正しい。

### 「なぜ」の補足（検討中）

テストコード自体は「何が・どの条件で・どう動くか」を表現する。「なぜそうなのか」を補う手段として以下を検討:

- **コメント / メタデータ**: テストコード内にメタデータ的なコメントを記述する（RBS inline 的な記法など）
- **ヘルパーメソッドによる拡張**: テストフレームワークの拡張で「なぜ」を書く場所を提供する（理想的だが保守コストとのトレードオフ）

## アーキテクチャ

```
RSpec テストコード → [静的解析] → IR（中間表現） → [生成] → 仕様書（静的サイト）
```

3 つのレイヤーで構成され、それぞれが抽象化されている:

### 1. 静的解析レイヤー（RSpec → IR）

テストコードを AST レベルで解析し、構造化された中間表現（IR）に変換する。

- テストを実行せずにコードを解析する
- テストの階層構造（describe / context / it）、説明文、コメント、メタデータの抽出
- 現時点では RSpec のみ対応。IR が抽象化レイヤーとなるため、将来的に他フレームワーク対応は IR の手前を追加するだけで済む

### 2. IR（中間表現）

テストフレームワークにも出力形式にも依存しない、共通のデータ構造。

- テストの階層構造、説明文、メタデータ、補足情報を保持する
- 入力側（テストフレームワーク）と出力側（ドキュメント形式）を分離する抽象化レイヤー

### 3. ドキュメント生成レイヤー（IR → 静的サイト）

IR から、人間が読みやすい Web ドキュメントを生成する。

- テストの階層構造をそのまま仕様書のセクション構造に反映
- ナビゲーション・検索が可能な静的サイト
- 開発中はホットリロード対応、ビルドすれば静的サイトとしてデプロイ可能

## 第一段階の方針

- **RSpec のみ対応**する。一本道で RSpec → IR → Web UI を通し、各レイヤーの設計を固める
- 複数フレームワーク対応は、レイヤー間の抽象化が確立した後に拡張する

## 名前の由来

**Spekta** — 「スペクタ」はラテン語の *spectare*（観察する、見る）に由来し、英語の *spec*（仕様）と *spectra*（スペクトル、光の分解）を連想させます。テストコードという光をプリズムのように分解し、仕様書として可視化する、というコンセプトを表現しています。
